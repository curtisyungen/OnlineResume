Why go from developing planes to developing websites?

I never planned to work at Boeing. In fact, I didn't plan to be a Civil Engineer. I chose engineering as my college major because I thought it would be more like architecture. It wasn't. But I stuck with it anyway since it was fairly interesting and engineering is a strong field. Boeing selected me for an internship and later offered me a job which I accepted.

Although my work at Boeing is respectable and worthwhile, it's not something I'd be excited to go home and spend extra time doing. Programming, on the other hand, has ensnared my interest. I have no trouble spending an entire evening after my day job working on programming. Often I have to force myself to stop programming because it's past midnight and I have to wake up at 5:45am for work.

I know the tech industry is booming and programming is really popular nowadays, and all that's great. But what matters most to me is not the money or opportunities or the nap pods I hear many tech companies provide. What's important to me is being able to dedicate my time to something that I love doing. And over the past year I've discovered that thing that I love to do is programming.


How is my engineering experience applicable to the tech industry?

At Boeing, a good Design Engineer is diligent, thorough, and organized. They know how to approach a complex problem in an orderly and logical way and develop an optimized solution. They are resourceful, they test their ideas, and they verify that the results are correct and that all potential pitfalls have been mitigated. On top of this, they communicate clearly, collaborate with others, and adhere to common design standards and engineering specifications.

A good programmer, as I see it, possesses a similar skillset to that of an engineer. They must be organized and attentive to details. They must approach difficult problems strategically and develop a logical plan before writing any code. They must test their code and ensure that all bugs are fixed and vulnerabilities handled. Their code must be readable, work well in teams, and incorporate coding guidelines and best practices.

I've spent the last 6.5 years working on this skillset. While I am continually improving I believe that the skills I've developed here at Boeing will serve me well as I transition into the tech industry as a Full-Stack Web Developer.

In five years I'd like to be a Full-Stack Web Developer in the tech industry. I'd prefer to be at a larger corporation, but I always try to remain flexible and open-minded in my career path and know I can find enjoyment in almost any programming work I'm doing.

I also see myself continuing to develop my programming skills by pursuing a Master's degree in Computer Science. This is a path I started down in 2018 but discontinued due to my exciting discovery of the UW Coding Boot Camp. However, I have always intended to return to my CS classes after the camp was over.

As for location, I could see myself remaining in Seattle. Yet at the same time I've always thought I'd leave the Pacific Northwest at some point. Wherever I end up, I'm open to adventure and new challenges and I look forward with optimism and enthusiasm to whatever the future may bring.

My Approach to the Sudoku Project

DESCRIPTION

  • This is a personal project that I started in March 2019.
  • It's not finished yet, so here's how I'm approaching it.

RULES

  • Fill out 9x9 grid such that each row, column, and 3x3 section has a unique set of numbers 1-9.

GENERATE NEW 9X9 SUDOKU PUZZLE

  • There are 81 total cells.
  • Each cell maintains its own array of "allowed numbers", initially 1-9.
  • Loop through grid cell by cell.
  • For each cell choose a random number from cell's Allowed Numbers array.
  • Validate that this number meets game's rules -- no duplicates in cell's row, column, or section.
  • If valid, insert chosen number into cell.
  • If not valid, delete number from cell's Allowed Numbers array and choose another random number from this array.
  • If cells Allowed Numbers array becomes depleted, replenish its Allowed Numbers array and then backtrack to previous cell. In previous cell, remove current value from Allowed Numbers array and choose a new random number.
  • Backtracking will be done as many times as needed until 9x9 grid is completely filled out to meet game rules.

DISPLAY BOARD ON SCREEN

  • Use generated puzzle to populate board on screen.
  • Populate cells at random.
  • Leave a set amount of cells blank depending on chosen difficulty.

USER SOLVES PUZZLE

  • With each keystroke compare user's input against puzzle solution.
  • If user's input doesn't match solution, provide feedback, i.e. change cell color.
  • If user submits solved puzzle, compare against solution. If valid, save total time to solve.

POTENTIAL CHALLENGES

  • Could there be more than one solution to a generated puzzle?

My Approach to the Chess Project

DESCRIPTION

  • This is a personal project I started in March 2019.
  • It's not finished yet, but here's how I'm approaching it.

GENERATE CHESS BOARD

  • Create an 8x8 empty board.
  • Assign a coordinate to each square based on its row x col position.
  • Assign a value to each cell based on what piece it holds.

GENERATE CHESS PIECE ARRAY

  • Generate white and black pieces.
  • Assign value to each piece and store in piece array.
  • For example: white pawn = 1, white rook = 2, white knight = 3, black pawn = 7, black rook = 8, black knight = 9
  • Empty square is assigned value of 0.
  • Display pieces in starting position on board.

HANDLE PIECE MOVEMENT

  • For each piece, keep track of legal moves it can make from its current square.
  • Store these legal moves in a Legal Moves array.
  • When user moves piece, check that this square is in this piece's Legal Moves array.
  • If valid, move piece.
  • If invalid, reset piece to current position.

HANDLE PAWN PROMOTION

  • If pawn reaches farthest rank of opposing side, allow user to promote.
  • Update square value depending on piece selected by user.

KEEP TRACK OF IN-GAME POSITION

  • Get values from each square indicating which piece they hold.
  • Store these numerical values in an array.
  • Update array with each new move.

HANDLE PIECE CAPTURE

  • After each move, check value in destination square.
  • If square's value is greater than 0, this is a piece capture.

TRACK WHEN KING IS IN CHECK / CHECKMATE

  • If Legal Moves array of piece includes square of opposing side's king, king is in check.
  • If the king has no legal moves, and none of his comrades have legal moves that can block the check, this is checkmate.
  • Check if opposing player's attacking piece can be captured to remove threat.

DETERMINE IF DRAW GAME / STALEMATE

  • Check combination of pieces remaining on board to see if it meets stalemate criteria.
  • For example: two kings remaining, one king and one bishop remaining, one king and one knight, etc.

GENERATE CHESS TACTICS PUZZLES

  • Each tactics puzzle will be generated as a mid-game position on the board.
  • Each puzzle is stored as a numerical array in the database.
  • In each puzzle, the goal is to either force checkmate or take advantage of opponent's blunder.
  • The moves are pre-defined and there is only one correct set of moves to make for each puzzle.
  • Check user's chosen move against correct move in puzzle. Give feedback if incorrect.
  • Complete puzzle if user executes correct move sequence on first try.
  • Display answer if incorrect.
  • Track user's progress -- number correct, number incorrect, consecutive puzzles solved correctly, etc.

POTENTIAL CHALLENGES

  • This is going to be a lot of juggling arrays which might be confusing at times.
  • All in all this will be a substantial undertaking.
